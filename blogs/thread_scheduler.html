How Fast Can a Context Switch Go?

July 2026

I'm building a user-level thread scheduler — cooperative threads that live entirely in userspace, with no kernel involvement, that hand control to each other explicitly instead of being preempted by a timer. The core of that is a function that swaps which thread the CPU is currently running: save one thread's state, load another's, keep going. That function is maybe ten lines of assembly.

I got it working, ran it once just to see a number, and got nerd-sniped. 11.51 nanoseconds per switch. Is that fast? Slow? What's actually taking that long? I put the scheduler down and spent the next while chasing that one number instead — trimming registers, fixing a broken benchmark, and eventually hitting a CPU exception I still can't fully explain.

This post is that detour, not the scheduler. If you're new to this level of the stack, I'll explain the concepts as they come up — you don't need to have written assembly before to follow along.

What actually needs to move

Every running program has some state that lives entirely inside the CPU, not in memory — a handful of registers it's using as scratch space, plus one register that points at where the next instruction lives and another that points at the current stack. A context switch is just: write all of that down somewhere, load a different program's saved values into those same registers, and carry on. The CPU doesn't know or care that it just became a different thread — it only ever sees "next instruction, current registers."

That sounds like it should involve a lot of state to save. In practice, for two cooperative user-level threads that never touch floating point, it's almost nothing. Here's the x86-64 register file and what each one is for:

Register Role Do I need to save it?
rip instruction pointer — where execution resumes yes
rsp stack pointer — which stack we're running on yes
rbp frame pointer (by convention, not hardware-enforced) yes, callee-saved
rbx general purpose, callee-saved yes, callee-saved
r12–r15 general purpose, callee-saved only if you actually use them
rflags condition codes — carry, zero, sign, overflow, etc, set by arithmetic/compare instructions and read by conditional jumps almost never, see below
rax, rcx, rdx, rsi, rdi, r8–r11 caller-saved (argument/return/scratch registers) no — the C calling convention already guarantees these are dead across a function call
xmm0–xmm15 floating point / SIMD only if the switched code touches floats
fs thread-local storage base only if you have per-thread TLS

Why rflags almost never needs saving: it's volatile on a timescale of single instructions. Any arithmetic or compare instruction overwrites it, and the C calling convention makes no promise that it survives a function call — the compiler never assumes flags are live across a call, because it has no way to know what the callee did internally. By the time ctx_switch returns into either thread, whatever code runs next is going to execute its own compare or arithmetic instruction before it branches on flags for anything meaningful, which overwrites them anyway. The one case where you'd actually need pushfq / popfq here is if you're switching in the middle of an instruction sequence that depends on a flag set by something that hasn't executed its next branch yet — which cooperative, function-call-boundary switching by construction never does, since ctx_switch is only ever entered and exited at a call/return boundary, never mid-sequence.

That caller-saved/callee-saved distinction is the one that makes the whole thing cheap: caller-saved registers don't need to be part of the context at all. ctx_switch is just a normal C function as far as the calling convention is concerned. Any caller that had something live in rax or rdi across the call already knows the compiler would have clobbered those anyway — that's what "caller-saved" means. The compiler on the calling side has already spilled anything it cared about before the call instruction executes. We only owe the callee-saved set: rbx, rbp, r12–r15.

So a minimal switch between two threads that only run integer code needs:

  • rip and rsp — because these define what "the thread" even is
  • rbp, rbx — callee-saved, ABI obligation
  • r12–r15 — callee-saved, but only if something is actually using them

That's it. No rax, no rflags in most cases, no FPU/SSE state unless the switched code does floating point. Here's the struct that holds it: 

typedef struct context_t {
    /*
    rax 
    rip -> instruction pointer where the task resumes.
    rbx -> calle-saved register; optinally used as base pointer
    rcx -> used to pass 4th arg to func, loop counter
    rdx -> used to pass 3rd arg to func, 2nd return register for 128bit returns (rdx:rax)
    rsp -> stack pointer
    rbp -> calle-saved register; optionally used as frame ptr
    rsi -> used to pass 2nd arg to func; source index
    rdi -> used to pass 1st arg to func; destination index
    r8 -> used to pass 5th arg to func
    r9 -> used to pass 6th arg to func
    r10 -> tmp reg; used to passing func;s static chain ptr
    r11 -> tmp register
    r12-r15 -> calle-saved register
    xmm0-xmm1 -> used to pass & return floating point arg
    xmm2-xmm7 -> used to pass floating point arg
    xmm8-xmm15 -> tmp register
    mmx0-mmx7 -> tmp register
    st0,st1 -> tmp register; used to return long double arg
    st2-st7 -> tmp register;
    fs -> reserved for system(as thread specific data reg)
    mxcsr -> sse2 control and status word
    x87 sw -> status word
    x87 cw -> control word
    */
    uint64_t rip; // where to resume
    uint64_t rsp; // stack top
    uint64_t rbp; // frame base
    uint64_t rbx; // callee-saved
} context_t;

The switch itself

.globl ctx_switch
.globl ctx_switch
.type ctx_switch, @function
ctx_switch:
    /* rdi -> old context */
    /* rsi -> new context */
    /* the `call` instruction that invoked this function pushed
    the return address (rip) onto the stack. we read it from (%rsp)*/
    movq (%rsp), %rax
    movq %rax, 0(%rdi) /* rip */

    leaq 8(%rsp), %rax
    movq %rax, 8(%rdi) /* rsp: save pre-call rsp (rsp+8), not rsp which points AT the return addr */

    /*movq %rsp, 8(%rdi) rsp */
    movq %rbp, 16(%rdi)/* rbp */
    movq %rbx, 24(%rdi) /* rbx */

    /* load the context to the regis */
    movq 24(%rsi), %rbx
    movq 16(%rsi), %rbp

    movq 8(%rsi), %rsp

    pushq 0(%rsi)
    ret

Two things worth slowing down on, because they're the whole trick:

Saving rip without an explicit "get current instruction pointer" instruction. x86-64 doesn't have one in user mode. But ctx_switch was reached via call ctx_switch, and call pushes the return address onto the stack before jumping. So (%rsp) — the top of the stack, right now, before we touch anything — already is the address we need to resume at. We just read it back out.

Resuming via pushq+ret instead of jmp. Once rsp has been swapped to the other thread's stack, we push that thread's saved rip onto its own stack and ret. ret pops an address and jumps to it — so this behaves exactly like returning from a function call, except the "function" we're returning into is wherever that thread was last suspended. The alternative — loading rip into a register and jmp *%rax — works too, but skips the CPU's return-address predictor (the RSB), which is tuned specifically for the push-then-ret pattern. Every real coroutine library (boost.context, libco, etc.) uses this exact trick for that reason.

One asymmetry to notice: the first time a thread runs, there's no "suspended state" to resume — it needs to start at a function entry point, not return into the middle of one. That's what init_task sets up: it manually writes a context_t that looks like what ctx_switch would have saved if the thread had been suspended one instruction before its own start, sitting on a stack laid out to match. 

void init_task(context_t* ctx, void (*func)(), void* stack, size_t stack_size) {
    uint64_t* stack_top = (uint64_t*)((uint8_t*)stack + stack_size);
    stack_top = (uint64_t*)((uintptr_t)stack_top & ~0xF);  // 16-byte align
    stack_top--;
    *stack_top = 0;              // dummy return address; func() never returns

    ctx->rip = (uint64_t)func;
    ctx->rsp = (uint64_t)stack_top;
    ctx->rbp = 0;
    ctx->rbx = 0;
}

void task_function() {

    }

uint8_t task_stack[4096];

void init_task(context_t* ctx, void (*func)(), void* stack, size_t stack_size) {
    uint64_t* stack_top = (uint64_t*)((uint8_t*)stack + stack_size);
    stack_top = (uint64_t*)((uintptr_t)stack_top & ~0xF);
    stack_top--;
    *stack_top = 0;

    ctx->rip = (uint64_t)func;
    ctx->rsp = (uint64_t)stack_top;
    ctx->rbp = 0;
    ctx->rbx = 0;
    // ctx->rflags = 0x202;
}

// static volatile int stop = 0;
static int stop = 0;

void alarm_handler(int sig) {
    stop = 1;  // set flag on SIGALRM, main loop checks it
}

int main() {
    init_task(&task_ctx, task_function, task_stack, sizeof(task_stack));

    signal(SIGALRM, alarm_handler);
    alarm(1);  // fire SIGALRM after 1 second

    long long count = 0;

    while (!stop) {
        ctx_switch(&main_ctx, &task_ctx);
        count++;
    }

    // task_function is still alive in its infinite loop, no need to touch it

    printf("Round trips (main<->task) : %lld\n", count);
    printf("Total context switches    : %lld\n", count * 2);
    printf("Avg per switch            : %.2f ns\n", 1e9 / (double)(count * 2));

    return 0;
}
//taskset -c 0 ./main

The first numbers

Wired up between a main loop and a task that just switches straight back, round-tripping as fast as possible for one second: 

Round trips (main<->task) : 43457474
Total context switches    : 86914948
Avg per switch            : 11.51 ns

11.51 ns. On a modern core that's somewhere around 30–40 cycles for what is, at the assembly level, eight instructions and two branches (the call on the way in, the ret on the way out). That felt slow enough to be worth questioning — and shallow enough as a benchmark that the number turned out to be measuring the wrong things almost as much as it was measuring the switch.

Asking the CPU directly

Instead of guessing at what those 30-odd cycles are doing, perf stat will just tell us — it reads the CPU's own hardware performance counters while your program runs, so instead of a stopwatch number you get things like "how many instructions actually retired" and "how often did a branch prediction turn out wrong." 

perf stat -e cycles,instructions,branches,branch-misses taskset -c 0 ./main

Total context switches    : 87128812

First thing to notice, if you haven't seen this before: every counter is split into cpu_atom and cpu_core. That's not a perf quirk — it means this is a hybrid CPU(i am on intel i5-13450hx) with two different kinds of cores on the same die (Intel calls them P-cores and E-cores; "atom" and "core" are perf's names for the same split). taskset -c 0 pins the process to logical CPU 0, but it doesn't tell you which physical core type that maps to, and the counter names are telling you: almost everything landed on cpu_core (the 99.96% next to each line is perf's confidence that the count is complete and not multiplexed away) — so this run executed on a P-core, and the tiny cpu_atom numbers are noise from something else briefly scheduled onto an E-core during the same second. Worth knowing before trusting any single-run number on a hybrid chip: which core type you land on isn't guaranteed just because you picked "CPU 0."

Taking the cpu_core row as the real numbers: 1.44 billion instructions in 3.76 billion cycles is an IPC (instructions per cycle) of about 0.38. That's low — a modern out-of-order core can retire multiple instructions per cycle when things are going well, so 0.38 means the pipeline is spending most of its time stalled, not executing.

The branch numbers explain why. 87,167,146 branch misses out of 261,526,087 branches is a 33% misprediction rate — roughly one in three branches guessed wrong. For a tight loop with only two branches per switch (the call into ctx_switch, and the ret back out), that's a strong signal that the RSB — the small stack of predicted return addresses the CPU keeps internally — is being defeated. Every switch rets to a different place than the call that would normally pair with it (main calls in, but the code path that eventually returns is on the task's own stack, and vice versa), so the predictor's assumption that call and ret nest like a normal function stack breaks down exactly the way you'd expect once you're bouncing between two independent stacks instead of staying on one.

4,449 L1 data-cache misses out of 568 million loads is effectively zero — about 0.0008%. That matches the earlier expectation: the whole context_t struct fits in a single 64-byte cache line, both structs together are a few hundred bytes, and that working set stays resident in L1 permanently across 87 million iterations. Same story for the instruction cache and the TLB — both negligible. Whatever is costing these ~43 cycles a switch, it isn't memory. It's the branch predictor losing its footing on every single ret.
